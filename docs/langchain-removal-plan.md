# LangChain / LangGraph removal — Phase 10 plan

**Status:** scheduled, not started.
**Origin:** 2026-05-25 conversation. The user noted that Phase 9 (AgentKit
migration) was supposed to remove LangChain/LangGraph entirely but
stopped at the 7 main autonomous agents, leaving 8 files still on the
old stack. This plan covers the remaining work.

**Why this is "Phase 10" and not part of Phase 9:** Phase 9's commit
messages explicitly say "7 agents". Whoever did the migration scoped to
the autonomous agent crons and consciously left workflows + the 8th
agent + tool-definition helpers for later. "Later" never happened — same
pattern as the Neo4j removal that lingered for weeks after Graphiti+
FalkorDB landed. This plan exists so Phase 10 doesn't drift the same way.

---

## What still uses LangChain / LangGraph

Eight files across three categories. Generated by:
```
grep -rln 'from "@langchain/' artifacts/api-server/src --include="*.ts"
```

### Category A — LangGraph state-machine workflows (5 files)
These are branching DAGs with checkpointing (LangGraph's `StateGraph`).
AgentKit can replace them with Networks, but it's a real conversion per
workflow — not a search-and-replace.

| File | Purpose | Cron / trigger | Risk if migration breaks |
|------|---------|----------------|---|
| `services/enrichment/graph.ts` | Capability enrichment pipeline (Perplexity → synthesis → DB write) | Inngest `auto-enrich` cron + manual `/api/enrich/...` triggers | Enrichment stops; new caps don't get researched |
| `services/vcr/graph.ts` | Vendor / Capability Research pipeline | Manual VCR triggers + scheduled refresh | VCR runs fail |
| `services/bots/workflows/pe-weekly-diligence.ts` | PE persona weekly diligence cycle | Inngest cron `0 9 * * 1` | Weekly PE digest doesn't get generated |
| `services/bots/workflows/cross-bot-system.ts` | Multi-persona cross-talk loop | Inngest event-driven | Bot conversations stall |
| `services/bots/workflows/persona-cycles.ts` | Per-persona think-act-reflect loop | Inngest cron per persona | Persona work stops |

### Category B — Lone LangChain agent (1 file)
| File | Purpose | Migration shape |
|------|---------|---|
| `services/disruption-vector-agent.ts` | The 8th autonomous agent. Uses `runReactAgent` from `services/agent/base-agent.ts`. Phase 9 migrated 7 of 8 agents; this one was skipped. | Same shape as the 7 Phase-9 migrations — wrap as an AgentKit Agent inside a Network. |

### Category C — Tool-definition helpers (2 files)
These use `tool()` from `@langchain/core/tools` to define tool wrappers
with Zod schemas. The migrated AgentKit agents currently call these via
the LangChain `.invoke()` interface that `tool()` provides.

| File | Why it stayed | What replacing it costs |
|------|----------------|---|
| `services/agent/tools.ts` | Every Phase-9 AgentKit agent imports tools from here | Need to either (a) rewrite every tool as an AgentKit-native tool, OR (b) write a thin adapter that exposes `.invoke()` without needing `@langchain/core` |
| `services/vcr/tools.ts` | Used by `services/vcr/graph.ts` and possibly other consumers | Same as above |
| `services/agent/base-agent.ts` | Hosts `runReactAgent` for the Category-B agent | Goes away naturally once Category B is migrated |

## Recommended migration order

**1. Category B first (smallest, most isolated)**
- Migrate `services/disruption-vector-agent.ts` to its own
  `services/disruption-vector-agent-agentkit.ts`, mirroring the Phase-9
  pattern (`cvi-agent-agentkit.ts` is the reference).
- Once this is done, `services/agent/base-agent.ts` is deletable —
  nothing else imports `runReactAgent`.
- **Estimate:** 2-3 hours. **Risk:** low — single agent, isolated cron.

**2. Category C tool helpers — write a tiny adapter**
- Don't rewrite the tool definitions; write a `lib/tool-adapter.ts` that
  exposes a `tool()` function with the same `.invoke()` shape but with
  zero LangChain dependency. The AgentKit agents call `.invoke()` already
  so they don't need to change.
- This unblocks dropping `@langchain/core` from `package.json` once
  Category A is also done.
- **Estimate:** 1-2 hours. **Risk:** low — pure refactor.

**3. Category A workflows (the bulk of the work)**
- One workflow at a time. For each:
  1. Read the `StateGraph` definition and enumerate the nodes + edges.
  2. Build an AgentKit Network with one Agent per LangGraph node.
  3. Replace state-channel access with AgentKit's `NetworkState`.
  4. Replace `.invoke()` on the graph with the Network equivalent.
  5. Smoke-test in dev (with a real trigger) before deploying.
- **Order within Category A:** start with the smallest workflow first
  (likely one of the bot workflows), use it as the pattern reference for
  the larger enrichment + VCR conversions.
- **Estimate:** 4-6 hours per workflow × 5 = 20-30 hours total. **Risk:**
  medium per workflow — these are load-bearing pipelines, a bad
  conversion silently breaks a feature.

**4. Cleanup**
- Remove `@langchain/anthropic`, `@langchain/core`, `@langchain/langgraph`,
  `langchain`, `langsmith` from `artifacts/api-server/package.json`.
- Remove `LANGCHAIN_TRACING_V2` / `LANGCHAIN_API_KEY` / `LANGCHAIN_PROJECT`
  references from CLAUDE.md (they no longer affect anything).
- Update CLAUDE.md "Observability" section: previously said LangSmith
  still applied to the "remaining LangChain consumers"; that's now empty.

## What this is NOT
- Not a rewrite of the API contracts. The HTTP routes, event triggers,
  and downstream DB writes all stay identical.
- Not a "switch to Vercel AI SDK" project. AI SDK already handles all
  one-shot structured-output calls; this plan is only about the stateful
  agent / state-machine workloads.
- Not urgent. Nothing is broken or wasting money on LangChain — the
  packages are ~$0 ongoing cost (bundle weight only). The reason to
  do this is consistency + reducing the surface area someone has to
  understand to work on the codebase.

## What this IS
- The closing-out of Phase 9. Once done, the entire agent / workflow
  layer runs on a single framework (AgentKit). No more "is this one on
  the old stack or the new?" questions.
- Removal of ~50MB of node_modules.

---

## When to do it
Best after:
- The Perplexity caller migration (commit `07a7e1c`) has stabilized and
  the cost trace shows the cache is finally covering all paths.
- The Neo4j teardown (commit `25d5e7e`) is operationally finished on
  Railway (env vars unset, service deleted).

Worst case timing: while a workflow is actively in flight. Migrating
`enrichment/graph.ts` while someone is running an enrichment job is
asking for a half-completed job.

## How long it will take, end to end
Realistic: 1-2 focused sessions of 4-6 hours each. NOT a "few hours
in one go" — Phase 9 took the original author multiple days for the 7
agents, and this is essentially Phase 9's leftover plus a bit more.

## How to schedule it
Don't put this on a cron. Don't bundle it with unrelated work. When
you want it done, say "do the Category B + C migration" first as a
contained 4-hour session. After that lands and stabilizes, schedule
Category A as a multi-session sequence (one workflow per session).
