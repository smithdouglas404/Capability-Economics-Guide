## 1. System Overview & Theoretical Foundations

Inflexcvi is a full-stack intelligence platform that autonomously researches, scores, and advises on the economic value of organizational capabilities across industries. The platform operationalizes the theoretical work of Prahalad & Hamel (1990), Teece, Pisano & Shuen (1997), and the Resource-Based View tradition — transforming qualitative capability assessments into quantifiable economic signals.

The platform's central hypothesis: organizational capabilities are latent economic assets whose value can be inferred from triangulated external signals, modeled using Bayesian inference, tracked through time-series analysis, and benchmarked cross-sectorally using industry-adjusted GDP weighting.

The architecture solves four distinct hard problems simultaneously:

1. **Signal triangulation:** How do you construct a reliable point estimate for a capability score when ground-truth data doesn't exist? Answer: Bayesian consensus over heterogeneous sources.
2. **Temporal dynamics:** How do you model capability velocity when measurements are infrequent and noisy? Answer: Exponential Moving Average with adaptive decay.
3. **Cross-industry comparability:** How do you compare a capability score in Healthcare to one in Manufacturing when market conditions differ? Answer: GDP-weighted normalization with industry-specific maturity priors.
4. **Institutional memory:** How do you accumulate research insight across autonomous agent cycles without human intervention? Answer: Persistent semantic memory via Mem0 Cloud with typed memory classifications.

## 2. Architecture Principles

### 2.1 Design Axioms

1. **Agent-first data production** — All content (CVI scores, C-Suite perspectives, case study ROI) is generated by an autonomous AI agent. The frontend is a pure read surface. This eliminates human editorial latency and enables 3× daily refresh cycles.

2. **Source-grounded inference, not fabrication** — Every score, insight, and benchmark is grounded in live Perplexity web research (sonar-pro model) or cited sources from McKinsey, Gartner, Forrester, Deloitte, BCG, and Accenture. The platform does not hallucinate capability scores — it triangulates them from observable market signals.

3. **Persistent institutional memory with semantic indexing** — The agent accumulates typed memory (observation, pattern, insight, decision_context) across research cycles using Mem0 Cloud. Memory recall uses semantic similarity search, enabling retrieval of relevant prior context without explicit key-value lookup. This approximates the episodic-to-semantic memory consolidation observed in biological intelligence systems.

4. **Bayesian uncertainty quantification** — All capability scores carry explicit 95% credible intervals. Confidence is a first-class system attribute, not an afterthought. This is epistemically honest — the system knows what it doesn't know.

5. **Zero manual triggers in production** — The scheduler is fully autonomous. No human action is required to keep data fresh. Three trigger types: routine (8h interval), urgency watchdog (5min interval, conditional), and on-demand (via API).

6. **Type-system-enforced contract integrity** — Shared types between frontend and backend are generated from an OpenAPI 3.1 specification via Orval codegen. Types drift is architecturally impossible — if the backend changes its API contract, the frontend fails to compile.

7. **Semantic CSS token design system** — All UI colors use semantic tokens, never hardcoded Tailwind color values. This enables full theme portability and dark/light mode without component rewrites.

8. **Strict monorepo package boundaries** — Shared types, schemas, and database access are in dedicated library packages with explicit dependency declarations. Circular dependencies are compile-time errors.

### 3.1 Critical Design Decisions

**pnpm workspaces with catalog:** Shared dependency version pinning via `pnpm-workspace.yaml` catalog section. All packages reference `catalog:` for shared dependencies (React, Tailwind, Zod) — a single version bump in the catalog updates all packages atomically.

**TypeScript project references:** Each lib package has a `tsconfig.json` with `composite: true` and `declarationMap: true`. The root `tsconfig.json` references all libs. `tsc --build` at the root compiles incrementally, only recompiling changed packages. This is critical for CI performance as the monorepo scales.

**Orval codegen contract loop:** The canonical truth is `lib/api-spec/openapi.yaml`. Orval reads this and generates:
- `lib/api-client-react/` — typed TanStack Query hooks
- `lib/api-zod/` — typed Zod request/response validators

This creates a closed validation loop: frontend request shapes are validated by the same schemas that validate backend responses. Type divergence is a compile error.

**esbuild for API bundling:** The API server is bundled into a single `dist/index.mjs` file (~5MB) using a custom `build.mjs` script. External packages are bundled in — the deployment container only needs Node.js, not the full `node_modules` tree in production (though currently we ship node_modules for simplicity).

### 4.3 Authentication Model

The platform uses a lightweight session token model — randomly generated UUIDs stored in browser `localStorage`. This is an intentional design choice for the current single-tenant, self-service model:

- **No password authentication** — eliminates account recovery overhead and credential management surface area
- **No JWT** — eliminates cryptographic signing complexity and token refresh logic
- **UUID v4** — 2^122 entropy space; practically unguessable
- **Stateless validation** — each route validates the token against the DB directly; no session middleware required
- **Tradeoff:** Session tokens are non-revocable without DB deletion; acceptable at current scale

When multi-tenancy is required, this model upgrades to Clerk Auth (pre-built PKCE + JWT with org-level RBAC) with minimal code change.

### 4.4 Request Validation Pipeline

Every route follows a strict validation pipeline:

```
HTTP Request
    │
    ▼
Pino request logger (req ID, method, URL)
    │
    ▼
Zod schema validation (path params, query params, body)
    │  ├── On failure: 422 Unprocessable Entity + structured error array
    │  └── On success: typed parsed object forwarded to handler
    ▼
Route handler (async, Express 5 error propagation)
    │
    ▼
Drizzle ORM query (parameterized — no raw SQL from user input)
    │
    ▼
Zod response schema validation (optional, dev mode)
    │
    ▼
Pino completion logger (statusCode, responseTime)
```

Zod schemas are generated from the same OpenAPI spec that defines the frontend hooks. This means request validation is not hand-written — it is derived from the authoritative contract document.

## 5. Autonomous Agent Architecture

### 5.1 Theoretical Basis

The agent architecture is grounded in three theoretical frameworks:

1. **Active Inference (Friston, 2010):** The agent minimizes surprise about capability scores by acquiring information from Perplexity research. Low confidence scores trigger research (surprise-reduction behavior). This is mathematically analogous to variational free energy minimization, though implemented heuristically rather than formally.

2. **Bayesian Updating:** Prior beliefs about capability scores are updated with new evidence from heterogeneous sources. The posterior estimate serves as the next cycle's prior, creating a continual belief-refinement process.

3. **Exponential Smoothing for Velocity:** EMA with α=0.7 weights recent score deltas heavily while preserving historical signal. This approximates the behavior of a Kalman filter under stationary noise assumptions, without the computational overhead of full Kalman state estimation.

### 5.2 LangGraph State Machine

The agent is a stateful directed acyclic graph (DAG) built with LangGraph. The state object is typed and carries forward across all nodes:

```typescript
interface AgentState {
  runId: string;
  trigger: "startup" | "routine" | "urgency:*";
  memories: Memory[];                  // recalled at evaluate
  capabilities: CapabilityRecord[];    // loaded at evaluate
  researchTargets: string[];           // decided at decide
  researchResults: ResearchResult[];   // populated at research
  ceiComponents: CeiComponent[];       // computed at compute
  contentQueue: ContentTarget[];       // queued at decide
  errors: AgentError[];                // accumulated throughout
  stats: RunStats;                     // counters updated at each node
}
```

**Node execution graph:**

```
┌─────────────────────────────────────────────────────────────────┐
│                         AGENT RUN                               │
│                                                                 │
│  ┌──────────┐   ┌────────┐   ┌──────────┐   ┌─────────┐      │
│  │ evaluate │──▶│ decide │──▶│ research │──▶│ compute │      │
│  └──────────┘   └────────┘   └──────────┘   └────┬────┘      │
│                                                    │           │
│  ┌──────────┐   ┌──────────────────┐   ┌──────────▼────────┐ │
│  │ finalize │◀──│ generateContent  │◀──│    memorize       │ │
│  └──────────┘   └──────────────────┘   └───────────────────┘ │
└─────────────────────────────────────────────────────────────────┘
```

**Node Responsibility Matrix:**

| Node | Inputs | Outputs | Failure Mode |
|------|--------|---------|--------------|
| `evaluate` | DB capabilities, Mem0 memories | Staleness assessments, confidence flags | Continues with empty memories if Mem0 fails |
| `decide` | Staleness/confidence assessments | Research target list, content queue | Skips all research if 0 targets selected |
| `research` | Target capability list | 4 Perplexity results per capability | Partial failure: failed capabilities marked, others proceed |
| `compute` | Research results, current DB scores | Bayesian posteriors, new CVI snapshot | Rolls back on DB write failure |
| `memorize` | Research findings, decisions | Mem0 write confirmations | Non-blocking — failure logged, agent continues |
| `generateContent` | Content queue, Perplexity context | C-Suite perspectives, case study ROI | Per-item failure: failed items logged, others proceed |
| `finalize` | Run stats, all prior outputs | DB run record, SSE completion event | Best-effort — never blocks process exit |

### 5.3 Research Methodology: The 4-Source Protocol

For each capability targeted for research, the agent executes exactly 4 Perplexity queries using distinct epistemic frames:

```
Query 1: "Current state of [capability] in [industry] — consulting perspective 2025-2026"
         → Sources: McKinsey, BCG, Deloitte, Accenture engagements
         
Query 2: "Market adoption benchmarks for [capability] in [industry] 2025"
         → Sources: Gartner Magic Quadrant, Forrester Wave, IDC reports
         
Query 3: "Academic research on [capability] maturity measurement 2023-2026"
         → Sources: Peer-reviewed journals, working papers, MIT/HBS/Wharton research
         
Query 4: "Practitioner implementations of [capability] — case studies and outcomes"
         → Sources: CIO/CDO case studies, vendor whitepapers, conference proceedings
```

This multi-perspective design reduces source bias — no single consulting firm's framing dominates the score. The diversity of epistemic sources is the primary defense against systematic over- or under-estimation.

Each query returns a structured response including:
- Estimated score (0-100)
- Source confidence (0-1)
- Key evidence cited
- Trend direction (improving/stable/declining)
- Time horizon of evidence

### 5.4 Bayesian Consensus Model

**Mathematical formulation:**

Let $\theta_c$ be the true (unobservable) capability score for capability $c$.

**Prior distribution:**
$$p(\theta_c) = \mathcal{N}(\mu_0, \sigma_0^2)$$

where:
- $\mu_0 = 50$ (center of 0-100 scale — non-informative prior for first cycle)
- $\mu_0 = \hat{\theta}_{c,t-1}$ (previous cycle's posterior mean for subsequent cycles)
- $\sigma_0^2 = 625$ (SD=25, spans roughly 2 standard deviations across most of the 0-100 range)

**Likelihood:** For each source $i \in \{1, 2, 3, 4\}$, the observed score $x_i$ is modeled as:
$$p(x_i | \theta_c) = \mathcal{N}(\theta_c, \sigma_i^2)$$

where $\sigma_i^2 = (1 - \text{confidence}_i)^{-1} \cdot 100$ reflects source uncertainty.

**Posterior update (conjugate Gaussian):**
$$p(\theta_c | x_1, ..., x_n) = \mathcal{N}(\mu_n, \sigma_n^2)$$

$$\mu_n = \frac{\mu_0/\sigma_0^2 + \sum_i x_i/\sigma_i^2}{1/\sigma_0^2 + \sum_i 1/\sigma_i^2}$$

$$\sigma_n^2 = \left(\frac{1}{\sigma_0^2} + \sum_i \frac{1}{\sigma_i^2}\right)^{-1}$$

**Confidence score (system-level):**
$$\alpha_c = 1 - \frac{\sigma_n}{25}$$

bounded to $[0, 1]$. A posterior standard deviation of 0 gives confidence=1.0; a posterior SD of 25 (prior uncertainty) gives confidence=0.0.

**95% credible interval:**
$$CI_{95} = [\mu_n - 1.96\sigma_n, \; \mu_n + 1.96\sigma_n]$$

This interval is stored in `cei_components.confidence_interval` and surfaced in the UI as the uncertainty band on capability radar charts.

### 5.5 Capability Value Index (CVI) Formula

**Full formula:**

$$\text{CVI} = \frac{\sum_{c} W_c \cdot C_c \cdot (1 + V_c) \cdot E_c \cdot \alpha_c}{\sum_{c} W_c} \times 10$$

**Variable definitions and derivations:**

| Variable | Symbol | Range | Derivation |
|----------|--------|-------|------------|
| Industry GDP weight | $W_c$ | 0.01–0.35 | Capability's industry contribution normalized by sector GDP share (World Bank data) |
| Bayesian consensus score | $C_c$ | 0–100 | Posterior mean from 4-source Perplexity triangulation |
| Velocity (EMA) | $V_c$ | -0.5–+0.5 | $V_c = \alpha \cdot \Delta C_{c,t} + (1-\alpha) \cdot V_{c,t-1}$, where $\alpha=0.7$ |
| Economic multiplier | $E_c$ | 1.0–2.0 | Derived from ontology dependency graph: capabilities with more dependents receive higher multipliers |
| Confidence | $\alpha_c$ | 0–1 | Posterior precision normalized (see §5.4) |

**Scale interpretation:**
- 0–200: Nascent (pre-investment; significant structural gaps)
- 200–400: Developing (early adoption; inconsistent implementation)
- 400–600: Advancing (broad deployment; competitive parity)
- 600–800: Leading (differentiated capability; above-industry performance)
- 800–1000: Transformative (capability as competitive moat; industry-defining)

**Velocity semantics:** $V_c > 0$ means the capability is improving relative to its recent trend. The $(1 + V_c)$ term amplifies the contribution of improving capabilities and discounts declining ones. This creates an implicit momentum signal — capabilities on a strong upward trajectory are weighted more heavily than stagnant ones with identical current scores.

**Multiplier derivation:** The economic multiplier $E_c$ is computed from the ontology graph using a PageRank-inspired algorithm. Capabilities that are prerequisites for many other capabilities receive higher multipliers (they are "platform capabilities"). For example, Data Architecture (score: 72) that enables AI Operations, Fraud Detection, and Predictive Analytics receives a multiplier of ~1.8×.

### 5.6 Scheduler Architecture & Urgency Logic

```
Server boot
    │
    ▼
startScheduler()
    │
    ├─── executeRun("startup")                     [t=0, immediate]
    │         Runs full agent cycle on boot.
    │         Populates CVI if DB is empty.
    │
    ├─── setInterval(8 hours)                      [t=8h, 16h, 24h, ...]
    │    └── executeRun("routine")
    │         Full research cycle across all targeted capabilities.
    │
    └─── setInterval(5 minutes)                    [continuous watchdog]
         └── detectUrgentConditions()
              │
              ├── any capability confidence < 0.35?
              │   └── executeRun("urgency:low_confidence")
              │
              ├── any capability not researched in 10+ days?
              │   └── executeRun("urgency:stale_data")
              │
              └── CVI dropped > 5 points since last snapshot?
                  └── executeRun("urgency:cei_drop")
```

**Mutex pattern:** `isRunning: boolean` flag in module scope. `executeRun` checks and sets this flag atomically (JavaScript single-threaded event loop guarantees atomicity). Overlapping runs are impossible within a single process. Distributed mutex (Redis Redlock) is required when scaling to multiple API server instances.

### 5.7 Memory Architecture (Mem0 Cloud)

**Memory type taxonomy:**

| Type | Purpose | Example Content |
|------|---------|-----------------|
| `observation` | Raw findings from this research cycle | "Insurance fraud detection score improved from 71→78 after new ML vendor entrants Q1 2026" |
| `pattern` | Cross-cycle patterns requiring context | "Healthcare AI adoption consistently leads other industries by 8-12 points across 6 cycles" |
| `insight` | Strategic synthesis connecting multiple observations | "Claims automation velocity is accelerating; suggest increasing research frequency" |
| `decision_context` | Why specific research decisions were made | "Skipped underwriting research this cycle — confidence 0.82, researched 3 days ago" |

**Memory recall mechanics:** At the start of each `evaluate` node, the agent issues a semantic similarity query: "capability research context for [industry] [current date]". Mem0 returns the top-K most semantically relevant memories ranked by embedding cosine similarity. The agent injects these into the LangGraph state, making them available to all downstream nodes.

This architecture approximates the **dual-memory model** from cognitive neuroscience (Complementary Learning Systems theory): working memory (AgentState, persists for one cycle) + long-term memory (Mem0 Cloud, persists indefinitely). Over hundreds of cycles, the agent accumulates institutional knowledge about industry dynamics, capability relationships, and research quality that informs increasingly precise decisions.

**Memory growth trajectory:** Each cycle stores 3-8 memories. At 3 cycles/day: ~3,000 memories/year. Mem0's semantic search remains efficient at this scale. At 10,000+ memories, we would introduce memory consolidation — periodically merging `observation` memories into higher-level `pattern` memories using Claude.

### 5.8 Real-Time Agent Activity (SSE)

The dashboard's live agent panel connects to `/api/agent/events` — a Server-Sent Events stream persisting for the duration of user connection. Events are typed:

```typescript
type AgentEvent =
  | { type: "agent_started"; runId: string; trigger: string }
  | { type: "phase_changed"; phase: AgentPhase; runId: string }
  | { type: "research_completed"; capabilityId: number; score: number; confidence: number }
  | { type: "cei_updated"; newIndex: number; delta: number }
  | { type: "memory_stored"; memoryType: MemoryType; content: string }
  | { type: "agent_completed"; runId: string; stats: RunStats }
  | { type: "scheduler_started"; nextRunIn: number }
  | { type: "heartbeat" }   // every 30s to keep connection alive
```

Events are broadcast to all connected SSE clients via an in-memory subscriber list. At single-process scale, this works without a message broker. At multi-process scale, this requires a pub/sub mechanism (Redis pub/sub or Kafka) to broadcast events from whichever process runs the agent to all API server instances serving SSE connections.

## 6. Database Architecture

**Engine:** PostgreSQL 16  
**ORM:** Drizzle ORM (type-safe query builder + schema management)  
**Schema management:** `drizzle-kit push` (schema-diff based, applies incremental DDL)  
**Connection:** Single connection pool via `pg` driver (max 10 connections default)

### 6.1 Schema Design Philosophy

The schema is normalized to 3NF with deliberate denormalization for read-performance at the CVI dashboard layer. `cei_snapshots` and `cei_components` are intentionally redundant with `capabilities` — they are append-only time-series tables optimized for dashboard reads, not the canonical capability definition.

JSONB is used for `metrics` columns (capability KPIs, case study metrics) where the key set is dynamic and known only at agent generation time. This avoids wide, sparse tables with many null columns.

### 6.3 Indexing Strategy

## 8. AI Integration Architecture

### 8.1 Anthropic Claude

| Attribute | Development | Production |
|-----------|------------|------------|
| Access method | Replit AI Integrations proxy (no key required) | Direct Anthropic API |
| Key management | Proxy handles auth | `ANTHROPIC_API_KEY` env var |
| Model | claude-3-5-sonnet | claude-3-5-sonnet |
| Usage pattern | Structured JSON output | Structured JSON output |
| Rate limiting | Proxy-managed | Self-managed (60k TPM) |

**Structured output pattern:** Claude is prompted with a JSON schema in the system prompt. All outputs are parsed with Zod. On parse failure, the request is retried once with a schema correction prompt. This achieves ~98% first-attempt parse success.

**Usage contexts:**
- Capability Assessment analysis (per-assessment, user-triggered)
- C-Suite perspective generation (per agent cycle, cached 48h)
- Case study ROI generation (per agent cycle, cached 48h)
- On-demand insight generation (per-insight, cached indefinitely)

### 8.2 Perplexity Sonar

**Model selection:**
- `sonar-pro`: 4 queries per capability (CVI research cycles) — higher cost, deeper web search
- `sonar`: On-demand research endpoint — lower cost, adequate for ad-hoc queries

**Query budget per run:** Maximum 6 Perplexity calls per agent run (configurable). At 6 capabilities per industry × 6 industries = 36 calls/cycle maximum. Budget limits prevent runaway costs during urgency triggers.

**Perplexity vs. alternatives:** Perplexity Sonar is preferred over direct web scraping because:
1. Real-time web access without browser automation complexity
2. Citation tracking built-in (URLs returned with each result)
3. Structured response format with source confidence scores
4. Rate limits are well-documented and predictable

### 8.3 Mem0 Cloud

**Architecture:** Mem0 uses a vector database backend (LlamaIndex with Qdrant) for semantic memory storage. The `mem0ai` SDK handles:
- Text embedding (OpenAI `text-embedding-3-small`)
- Vector similarity search (cosine)
- Memory deduplication (prevents near-duplicate memories)
- Memory consolidation (automatically merges similar observations)

**Agent identity:** Fixed `user_id = "cei-agent"`. All memories belong to this single agent identity. In multi-tenant deployments, this would be partitioned by `orgId` or `agentInstanceId`.

**Local sync:** Memories are synced to the `agent_memories` DB table for searchability without Mem0 API calls. This enables SQL queries over memory content (e.g., "show all memories mentioning Healthcare AI") that Mem0's semantic search alone can't answer.

### 8.4 Letta (Stateful Agent Memory)

**Configuration:**
- Base URL: `https://letta.innume.com` (self-hosted)
- Model: `openrouter/anthropic/claude-3.7-sonnet`
- Agent ID: `agent-9835f350-f904-4f04-b5ae-da315a0a1d5e`

**Role in architecture:** Letta provides persistent memory blocks that survive agent process restarts. While Mem0 handles semantic/episodic memory, Letta maintains structural memory blocks (core beliefs, organizational context, research heuristics) that need to be loaded at agent initialization, not just recalled on-demand.

**Graceful degradation:** The Letta client uses lazy initialization with a health check. If the Letta server is unreachable, the agent logs a warning and continues without Letta memory blocks. This prevents a single external service from blocking agent execution.

## 10. Scalability & Evolution Path

### 10.1 Current Architecture Constraints

| Constraint | Current Limit | Trigger for Upgrade |
|-----------|--------------|---------------------|
| Concurrent users | ~500 (Node.js event loop) | SSE connection pool exhaustion |
| Agent runs | 1 at a time (in-process mutex) | Multiple tenants needing simultaneous runs |
| DB connections | 10 (pg pool max) | Query queue latency >100ms |
| Memory recall latency | ~200ms (Mem0 API round-trip) | >50 recall operations per cycle |
| Perplexity budget | 6 calls/run | Multi-industry expansion |

### 10.2 Multi-Tenant Evolution

```
Current (single-tenant)          Multi-tenant (Phase 2)
─────────────────────────        ─────────────────────────────────
session token → org              Clerk Auth → JWT → org RBAC
fixed agent_id                   agent_id per org
shared CVI index                 CVI index per org + industry
in-process scheduler             BullMQ + Redis job queue per org
single Mem0 user_id              user_id = orgId
```

### 10.3 Agent Scale-Out

When multiple concurrent agent runs are needed:

```
Current: API Server → in-process agent
Future:  API Server → BullMQ queue → Worker Service → agent

Worker Service (separate Railway service):
├── BullMQ worker (Redis-backed)
├── One worker process per concurrent run
├── Redis pub/sub for SSE event broadcasting
└── Database: same PostgreSQL, separate connection pool
```

### 10.4 Research Volume Scale-Out

At 12 industries × 12 capabilities × 4 Perplexity queries × 3 cycles/day = 1,728 Perplexity calls/day. At Perplexity's `sonar-pro` pricing (~$5/1000 queries), this is ~$8.64/day or ~$260/month. Caching strategies:

- **Source deduplication:** Cache Perplexity responses by (query_hash, date) with 24h TTL in Redis
- **Adaptive frequency:** Research capabilities proportionally to their volatility; stable capabilities (monthly) vs. volatile capabilities (daily)
- **Hierarchical research:** Industry-level market trend queries shared across all capabilities in that industry

## 11. Security Architecture

### 11.1 Threat Model

| Threat | Vector | Mitigation |
|--------|--------|-----------|
| API key exposure | Code or environment leak | All keys via env vars; Replit secrets in dev; Railway variables in prod |
| SQL injection | User input in queries | Drizzle ORM parameterized queries; no raw SQL from user input |
| XSS | Stored content from AI | Content served as API JSON, not rendered as raw HTML; React escapes by default |
| Session hijacking | Session token theft | Tokens are single-org; no privilege escalation possible; low-value target |
| Prompt injection | User-supplied text to Claude | System prompts instruct Claude to ignore instructions in user content; structured output schema limits damage |
| LLM output abuse | Malicious structured output | Zod validates all Claude outputs; malformed responses are rejected, not executed |
| CORS misconfiguration | Cross-origin requests | CORS restricted to known origins in production |
| Rate limiting absent | API abuse | Rate limiting on `/api/research` (the only expensive user-triggerable endpoint) |

### 11.2 Data Privacy Posture

- No PII stored — organizations are identified by session token + optional name only
- No user tracking — no analytics platform, no third-party scripts
- Perplexity queries contain capability names and industry names, not user data
- Claude receives assessment text — users should be informed if PII is entered
- GDPR compliance: session data deletable via `DELETE /organizations/:token`

## Appendix B: Key Formulae Reference

**Bayesian posterior mean:**
$$\hat{\theta}_c = \frac{\mu_0/\sigma_0^2 + \sum_i x_i/\sigma_i^2}{1/\sigma_0^2 + \sum_i 1/\sigma_i^2}$$

**EMA velocity:**
$$V_{c,t} = 0.7 \cdot (C_{c,t} - C_{c,t-1}) + 0.3 \cdot V_{c,t-1}$$

**Economic multiplier (PageRank-inspired):**
$$E_c = 1 + \frac{\sum_{j: c \in \text{deps}(j)} \text{strength}(j,c) \cdot E_j}{N_{deps}}$$

**CVI:**
$$\text{CVI} = \frac{\sum_c W_c \cdot \hat{\theta}_c \cdot (1 + V_c) \cdot E_c \cdot \alpha_c}{\sum_c W_c} \times 10$$

**Confidence:**
$$\alpha_c = \max\left(0, 1 - \frac{\sigma_{n,c}}{25}\right)$$
