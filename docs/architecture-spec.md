# Capability Economics Platform — Deep Architecture Specification

**Version:** 1.0  
**Date:** April 2026  
**Classification:** Internal Technical Reference

---

## 1. System Overview

Capability Economics is a full-stack intelligence platform that autonomously researches, scores, and advises on the economic value of organizational capabilities across industries. It combines a live data pipeline, a stateful AI agent, a Bayesian scoring engine, and an executive-grade frontend into a unified advisory system.

The platform operates on a single foundational insight: organizational capabilities have measurable economic value that can be tracked, compared, and optimized — and that value shifts over time as industries evolve.

---

## 2. Architecture Principles

1. **Agent-first data** — All content (CEI scores, C-Suite perspectives, case study ROI) is generated by an autonomous AI agent, not hardcoded. The frontend is a read surface only.
2. **Real sources, no fabrication** — Every score, insight, and benchmark is grounded in live Perplexity web research or 102+ cited sources from McKinsey, Gartner, Forrester, Deloitte, BCG, and Accenture.
3. **Persistent institutional memory** — The agent learns across research cycles using Mem0 Cloud. Patterns, observations, and decision context accumulate and improve future research quality.
4. **Zero manual triggers in production** — The scheduler is fully autonomous. No human action is required to keep data fresh.
5. **Theme-token design system** — All UI colors use semantic tokens, never hardcoded Tailwind values. The design is fully themeable.
6. **Monorepo with strict package boundaries** — Shared types, schemas, and database access are in dedicated library packages consumed by both frontend and backend.

## 4. Backend Architecture (`artifacts/api-server/`)

### 4.1 Runtime
- **Framework:** Express 5 (async error handling built-in)
- **Port:** 8080 (via `PORT` env var)
- **Process model:** Single Node.js process with background scheduler thread
- **Logging:** Pino (structured JSON, with pino-pretty in development)
- **Validation:** Zod on all routes via generated `@workspace/api-zod` schemas

### 4.2 Route Structure
```
/api
├── /industries          GET — list all 6 seeded industries
├── /capabilities        GET — capabilities with filters (industry, role)
├── /roles               GET — 8 C-Suite roles
├── /organizations       POST/GET/PATCH — org CRUD with session tokens
├── /capabilities/assess POST — bulk capability assessment upsert
├── /dashboard           GET — personalized gap analysis (role-filtered)
├── /projects            GET/:id — technology project catalog + impacts
├── /insights            GET/POST — AI-generated insights, leaderboard, ontology
├── /research            POST — on-demand Perplexity research endpoint
├── /cei
│   ├── /current         GET — live CEI composite index
│   ├── /history         GET — historical snapshots
│   ├── /components      GET — per-capability breakdown
│   ├── /methodology     GET — formula documentation
│   └── /refresh         POST — manual CEI recompute trigger
├── /agent
│   ├── /status          GET — scheduler state + last run stats
│   ├── /history         GET — run history
│   ├── /memories        GET — Mem0 Cloud memories
│   ├── /tools           GET — integration status (Mem0, Perplexity, Letta)
│   └── /events          GET — SSE stream for live agent activity
├── /csuite              GET — AI-generated C-Suite perspectives (from DB)
├── /csuite/:slug        GET — single role perspective
└── /case-study/:slug    GET — AI-generated case study ROI data (from DB)
```

### 4.3 Authentication & Sessions
- Session tokens are UUID strings stored in browser `localStorage`
- No password auth — lightweight session model for single-user org assessments
- Token passed as query param or header; validated per-route with Zod

### 4.4 Data Validation Pipeline
All incoming requests pass through Orval-generated Zod schemas:
1. Path params validated (e.g., `industryId` must be valid integer)
2. Query params validated (e.g., `roleSlug` must be known enum value)
3. Request body validated (e.g., assessment scores must be 0-100)
4. Validation errors return structured 422 responses

## 5. Autonomous Agent Architecture

### 5.1 LangGraph State Machine

The agent is a stateful directed graph built with LangGraph. Each research cycle traverses these nodes:

```
evaluate → decide → research → compute → memorize → generateContent → finalize
```

**Node responsibilities:**

| Node | Purpose |
|------|---------|
| `evaluate` | Load current CEI components, recall relevant Mem0 memories, assess staleness and confidence |
| `decide` | Determine which capabilities need research (staleness > 7 days, confidence < 0.6, volatility high) |
| `research` | Call Perplexity 4× per capability (consulting, market data, academic, practitioner) |
| `compute` | Run Bayesian consensus + CEI formula, store snapshot |
| `memorize` | Store observations, patterns, and decision context to Mem0 Cloud |
| `generateContent` | Generate C-Suite perspectives and case study ROI data using Perplexity + Claude |
| `finalize` | Emit completion event via SSE, log run summary |

### 5.2 LangChain Tools

Five tools are available to the agent via `@langchain/core` tool interface:

| Tool | Description |
|------|-------------|
| `perplexity_research` | Makes focused web research queries via Perplexity Sonar API |
| `query_database` | Read-only DB access for capabilities, scores, and history |
| `compute_cei` | Executes the full CEI formula with Bayesian consensus |
| `recall_memories` | Semantic search over Mem0 Cloud memories |
| `store_memory` | Writes observations/patterns/decisions to Mem0 Cloud |
| `generate_csuite_perspectives` | Perplexity research → Claude structured output for C-Suite content |
| `generate_case_study` | Perplexity benchmarks → Claude structured ROI data for case studies |

### 5.3 Scheduler Architecture

```
Server boot
    │
    ▼
startScheduler()
    │
    ├── executeRun("startup")         ← immediate first run
    │
    ├── setInterval(8h)               ← routine research cycle
    │   └── executeRun("routine")
    │
    └── setInterval(5min)             ← urgency watchdog
        └── detectUrgentConditions()
            ├── confidence < 0.35?  → executeRun("urgency:low_confidence")
            ├── stale > 10 days?    → executeRun("urgency:stale_data")
            └── CEI drop > 5pts?   → executeRun("urgency:cei_drop")
```

Overlapping runs are prevented via `isRunning` mutex flag.

### 5.4 CEI Formula

```
CEI = Σ(Wi × Ci × (1 + Vi) × Ei × αi) / ΣWi × 10
```

| Variable | Definition |
|----------|-----------|
| Wi | Industry GDP weight (normalized to 1.0 sum) |
| Ci | Bayesian consensus score (posterior mean from 4 Perplexity sources) |
| Vi | Velocity — EMA(α=0.7) of score delta across recent snapshots |
| Ei | Economic multiplier — derived from ontology dependency graph (1.0–2.0×) |
| αi | Confidence — based on source agreement, recency, and sample size |

**Bayesian consensus:**
- Prior: μ=50, σ²=625 (non-informative, center of 0-100 scale)
- Likelihood: Each Perplexity source provides a score estimate
- Posterior: Weighted average of prior and observed scores
- Output: Point estimate (posterior mean) + 95% credible interval

**Scale:** 0–1000, mapped to bands: Nascent / Developing / Advancing / Leading / Transformative

### 5.5 Memory Architecture (Mem0 Cloud)

Four memory types are stored per research cycle:

| Type | Content |
|------|---------|
| `observation` | Raw findings from this cycle (scores, trends) |
| `pattern` | Cross-cycle patterns detected (e.g., "Insurance fraud detection consistently improving") |
| `insight` | Strategic synthesis (e.g., "Healthcare AI adoption acceleration outpacing infrastructure") |
| `decision_context` | Why the agent chose to research or skip specific capabilities |

Memory is recalled at the start of each `evaluate` node run. Higher recall count = richer context for research decisions.

### 5.6 Content Generation Pipeline

For C-Suite perspectives and case study content:
1. **Perplexity query** — "Real [role] priorities for [industry] capability [name] in 2025-2026"
2. **Claude structured output** — takes Perplexity context, outputs typed JSON matching DB schema
3. **DB write** — stored in `csuite_perspectives` or `case_study_content`
4. **Staleness check** — content generated within 48 hours is skipped
5. **Frontend read** — pages fetch from `/api/csuite` and `/api/case-study/:slug` — zero on-demand generation

### 5.7 Real-Time Agent Activity (SSE)

The dashboard connects to `/api/agent/events` — a Server-Sent Events stream that delivers:
- `agent_started` / `agent_completed` lifecycle events
- `phase_changed` — node transitions in the LangGraph state machine
- `research_completed` — per-capability research results
- `cei_updated` — new CEI score computed
- `memory_stored` — Mem0 write confirmations
- `scheduler_started` — scheduler boot confirmation

**Engine:** PostgreSQL 16 (Replit managed / Railway PostgreSQL plugin)  
**ORM:** Drizzle ORM with type-safe query builder  
**Schema management:** `drizzle-kit push` (schema-diff based, no migration files)

### 6.1 Core Domain Tables

```
capability_metrics      (KPIs per capability)
capability_dependencies (inter-capability relationships)
c_suite_roles           (8 executive roles)
capability_role_mappings (role relevance per capability)
```

### 6.2 Organization Tables

```
organizations              (user orgs with session tokens)
organization_capabilities  (assessment scores, unique on org_id + capability_id)
```

### 6.3 CEI Intelligence Tables

```
cei_snapshots           (point-in-time overall CEI index with timestamp)
cei_components          (per-capability score, confidence, velocity, multiplier)
source_triangulations   (4 Perplexity source scores per capability per run)
```

### 6.4 Agent Tables

```
agent_runs     (run history: trigger, status, stats, timing)
agent_memories (local sync of Mem0 Cloud memories for DB queries)
```

### 6.5 Content Tables

```
csuite_perspectives  (AI-generated role × industry content, generated_at for staleness)
case_study_content   (AI-generated capability ROI data, metrics as JSONB)
```

### 6.6 Insights & Research Tables

```
capability_thresholds    (R/Y/G threshold definitions with source citations)
capability_insights      (AI-generated + seeded strategic insights)
industry_leaderboard     (company benchmark rankings per industry)
industry_white_papers    (curated research with URLs and metadata)
data_sources             (citation tracking: publisher, date, URL)
ontology_relationships   (capability-to-capability semantic relationships)
ontology_industry_adapters (industry-specific maturity models)
technology_projects      (6 projects: AI, App Mod, Mainframe, Data)
project_capability_impacts (capability uplift estimates per project)
project_executive_insights (CFO/CEO/CIO agenda items per project)
project_risks            (risk-of-inaction with severity and mitigation)
```

## 8. AI Integration Architecture

### 8.1 Anthropic Claude
- **Access:** Via `@workspace/integrations-anthropic-ai` — Replit AI Integrations proxy (no API key needed in development)
- **Production:** Direct Anthropic API via `ANTHROPIC_API_KEY`
- **Lazy initialization:** Client instantiated on first use to avoid startup failures
- **Usage:** Structured output generation (C-Suite perspectives, case study ROI, on-demand insights)
- **Model:** claude-3-5-sonnet (balance of quality and cost)

### 8.2 Perplexity
- **Access:** Direct REST API via `PERPLEXITY_API_KEY`
- **Model:** sonar-pro for deep research, sonar for lighter queries
- **Usage:** CEI research (4 queries/capability), content grounding, on-demand research endpoint
- **Rate limiting:** Max 6 Perplexity calls per agent run

### 8.3 Mem0 Cloud
- **Client:** `mem0ai` npm package with `MemoryClient`
- **API key:** `MEM0_API_KEY` in secrets
- **Memory types:** observation, pattern, insight, decision_context
- **Recall:** Semantic search by query string, returns scored results
- **User ID:** Fixed `"cei-agent"` — all memories belong to the agent identity

### 8.4 Letta (Optional)
- **Client:** `@letta-ai/letta-client`
- **Status:** Optional integration — auto-connects to local Letta server
- **Degradation:** Gracefully disabled when unavailable (HTTP 404 → logs warning, continues)
- **Purpose:** Stateful agent memory blocks for cross-session institutional context

## 9. Deployment Architecture

### 9.1 Replit (Development / Preview)
- Two workflows: `api-server` (port 8080) and `capability-economics` (port from `PORT` env)
- Path-based routing: frontend at `/capability-economics`, API at `/api`
- PostgreSQL managed by Replit

### 9.2 Railway (Production)
- Single service: `@workspace/api-server`
- Build: `pnpm install --frozen-lockfile && pnpm run build:deploy`
- Start: `pnpm run start`
- The `build:deploy` script: typechecks libs → builds frontend SPA → builds API server bundle
- Frontend SPA is built into `artifacts/capability-economics/dist/` and served as static files by Express
- PostgreSQL via Railway PostgreSQL plugin (auto-injects `DATABASE_URL`)
- Environment variables: `MEM0_API_KEY`, `PERPLEXITY_API_KEY`, `NODE_ENV=production`, `PORT=8080`

### 9.3 Build Pipeline
```
build:deploy
    ├── typecheck:libs          (tsc --build on lib packages)
    ├── capability-economics    (vite build → dist/public/)
    └── api-server              (build.mjs → esbuild → dist/index.mjs)
```

## 10. Scalability Considerations

**Current state (single-tenant):** One org per session, one agent instance, one DB.

**Path to multi-tenant:**
- Replace session tokens with proper auth (Clerk or Replit Auth)
- Partition agent memories by `userId` in Mem0
- Add org-level CEI snapshots (currently industry-level)
- Queue agent runs per org (BullMQ or similar)

**Path to higher research volume:**
- Move agent to dedicated worker process (separate Railway service)
- Replace `setInterval` scheduler with proper job queue (BullMQ + Redis)
- Cache Perplexity responses with 24h TTL to reduce duplicate queries
- Add read replicas for CEI dashboard queries

## 11. Security Posture

- No secrets in code — all API keys via environment variables / Replit secrets
- Zod validation on all API inputs — no raw DB queries from user input
- Session tokens are random UUIDs, not JWTs — no cryptographic attack surface
- Anthropic client accessed via Replit proxy in development — no key exposure
- No PII stored — orgs are anonymous until user adds a name
