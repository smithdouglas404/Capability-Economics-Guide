# Installing Mem0 + Letta on Railway

Step-by-step recipe for deploying the **Mem0** memory layer and the **Letta** stateful agent into a Railway project, wired into the api-server. Distilled from the war stories in [`railway-setup.md`](./railway-setup.md) so you don't have to re-discover the traps.

> **Audience:** someone new to Railway who needs to spin these two services up against this repo (`smithdouglas404/Capability-Economics-Guide`).
>
> **Time:** ~20 minutes if everything goes right. Mem0 build is ~5 min; Letta is ~30 s.

---

Six services total. This doc covers Mem0 + pgvector + Letta only — the api-server and Postgres are assumed already set up.

## 1. Tokens & secrets you'll need

Collect these **before** you start clicking around Railway. Generate the ones you don't already have.

| Token | What it's for | Where to get it | Notes |
|---|---|---|---|
| **`OPENAI_API_KEY`** | Mem0 calls OpenAI for both LLM extraction and embedding generation | https://platform.openai.com/api-keys → "Create new secret key" | Need a paid account with usage credits. **Required** for Mem0 to do anything. |
| **`OPENROUTER_API_KEY`** | Letta uses OpenRouter to register LLM model handles (e.g. `openrouter/anthropic/claude-3.7-sonnet`) | https://openrouter.ai/keys → "Create Key" | Without this, Letta boots but has zero model handles → every agent run fails with `NOT_FOUND: Handle … must be one of []` |
| **`PERPLEXITY_API_KEY`** | api-server uses Perplexity for research in the autonomous agent loop | https://www.perplexity.ai/settings/api → "Generate" | Optional but recommended; degrades gracefully if missing |
| **`MEM0_ADMIN_API_KEY`** | Auth token for Mem0's REST API. The api-server sends this as `X-API-Key`. | **Generate yourself.** Run `openssl rand -base64 32` or `python3 -c "import secrets; print(secrets.token_urlsafe(32))"` | Used twice: set as `ADMIN_API_KEY` on Mem0, and as `MEM0_API_KEY` on capabilityeconomics. Must match exactly. |
| **`MEM0_JWT_SECRET`** | Mem0 uses this internally to sign JWTs (for its `/auth/login` endpoint — we don't use that flow, but Mem0 still requires the var to start) | **Generate yourself.** `openssl rand -base64 48` | Anything strong. Different from `MEM0_ADMIN_API_KEY`. |
| **`LETTA_SERVER_PASSWORD`** | Auth for Letta's HTTP API. api-server sends this as `LETTA_API_KEY`. | **Generate yourself.** `openssl rand -base64 32` | Used twice: set as `LETTA_SERVER_PASSWORD` on Letta, and as `LETTA_API_KEY` on capabilityeconomics. |
| **`PGVECTOR_PASSWORD`** | Postgres password for the pgvector instance backing Mem0 | **Auto-generated by Railway** when you add the pgvector template | You'll never type this; Railway auto-wires it into the Mem0 service |
| `ADMIN_API_KEY` (api-server) | Auth for the api-server's own `/api/admin/*` routes (e.g. enrichment triggers) | **Generate yourself.** `openssl rand -base64 32` | Distinct from `MEM0_ADMIN_API_KEY` — different system, different purpose, just unfortunate name overlap |

Quick one-liner for all the secrets you need to generate:

```bash
echo "MEM0_ADMIN_API_KEY  = $(openssl rand -base64 32)"
echo "MEM0_JWT_SECRET     = $(openssl rand -base64 48)"
echo "LETTA_SERVER_PASS   = $(openssl rand -base64 32)"
echo "API_SERVER_ADMIN    = $(openssl rand -base64 32)"
```

**Save these values somewhere immediately** — Railway shows generated secrets only once if you let it auto-create them; if you generate them yourself you control storage. Drop them in a password manager or a `.env.notes` file you keep outside the repo.

## 2. Add pgvector (the vector store for Mem0)

This must exist **before** Mem0, because Mem0's env vars reference pgvector's auto-injected credentials.

1. Railway dashboard → your project → **+ New** → **Database** → **Add pgvector**
   (If you don't see pgvector as a database template, use **Deploy Template** → search "pgvector" → pick `pgvector/pgvector:pg18`.)
2. Wait ~30 s for status to flip to **SUCCESS**.
3. Click into the pgvector service → **Variables** tab. You should see Railway auto-generated `PGUSER`, `PGPASSWORD`, `PGDATABASE`, `PGHOST`, `PGPORT`, and a few reference variables like `${{Postgres.DATABASE_URL}}`. Leave these alone.

**Internal hostname Railway assigned:** `pgvector.railway.internal` (case-sensitive — match exactly when wiring Mem0).

---

## 3. Add Mem0

1. Railway dashboard → your project → **+ New** → **GitHub Repo** → select `smithdouglas404/Capability-Economics-Guide` → branch `main`.
2. Once Railway creates the service, **rename it to `Mem0`** (the api-server expects this exact name via `MEM0_BASE_URL=http://mem0.railway.internal:8000`). Service Settings → at top → edit name.
3. **Set the source root directory.** Service Settings → **Source** → **Root Directory** → enter **`mem0`** → Save. (This is the critical step that makes Railway use the libpq5-patched Dockerfile in our repo rather than mem0ai/mem0's broken one.)
4. **Variables tab — paste in all of these:**

```env
# Auth (use the values you generated in step 1)
ADMIN_API_KEY=<MEM0_ADMIN_API_KEY from above>
JWT_SECRET=<MEM0_JWT_SECRET from above>
AUTH_DISABLED=false

# LLM provider — Mem0 uses OpenAI for extraction and embedding
OPENAI_API_KEY=<your OpenAI key>

# Tunables (optional — these defaults match the Mem0 template)
MEM0_DEFAULT_LLM_MODEL=gpt-4.1-nano-2025-04-14
MEM0_DEFAULT_EMBEDDER_MODEL=text-embedding-3-small

# Storage paths
HISTORY_DB_PATH=/app/history/history.db
APP_DB_NAME=postgres

# Postgres connection (points at pgvector)
POSTGRES_HOST=pgvector.railway.internal
POSTGRES_PORT=5432
POSTGRES_DB=postgres
POSTGRES_USER=postgres
POSTGRES_PASSWORD=${{pgvector.PGPASSWORD}}

# Server settings
PORT=8000
PYTHONUNBUFFERED=1
PYTHONDONTWRITEBYTECODE=1
```

   The `${{pgvector.PGPASSWORD}}` syntax is Railway's reference-variable syntax — it auto-injects the password from the pgvector service, so they always stay in sync.

5. **Deploy.** Settings → at top → "Deploy" (or it may have auto-deployed when you set the source). Build takes ~5 minutes (apt + clone + pip install). Watch the **Deployments** tab.
6. **Verify it's alive.** Once status is `SUCCESS`, click the service → **Settings** → **Networking** → generate a public domain (TEMPORARILY, for testing) → open `https://<that-domain>/docs` in a browser. You should see Mem0's Swagger UI. Test auth:

   ```bash
   curl -H "X-API-Key: <MEM0_ADMIN_API_KEY>" "https://<public-domain>/memories?agent_id=test&limit=1"
   # Expect: {"results":[]} with HTTP 200
   ```

   **Don't use `Authorization: Bearer` — Mem0's self-hosted v2.x treats Bearer headers as JWTs.** Always use `X-API-Key`. (The Mem0 Railway template doc page incorrectly says to use Bearer. Believe me, not them.)

7. **Remove the public domain** when verification is done — Mem0 should only be reachable from inside the Railway private network.

## 4. Add Letta

1. Railway dashboard → your project → **+ New** → **GitHub Repo** → same repo, `main` branch.
2. Rename the service to `letta` (the api-server expects internal hostname `letta-2eot.railway.internal` — the `-2eot` suffix or whatever Railway assigns will appear in the service URL automatically).
3. Service Settings → **Source** → **Root Directory** → enter **`letta`** → Save.
4. **Variables tab:**

```env
LETTA_SERVER_PASSWORD=<LETTA_SERVER_PASS from step 1>

# Provider for LLM handles — without this, Letta boots but has zero
# model handles and every agent run fails with:
#   "NOT_FOUND: Handle openrouter/anthropic/claude-3.7-sonnet not found,
#    must be one of []"
OPENROUTER_API_KEY=<your OpenRouter key>

PORT=8283
```

5. Deploy. Build is fast (~30 s — just pulls `letta/letta:latest` image, which is amd64-compatible).
6. **Note the exact internal hostname Railway assigns.** Service Settings → **Networking** → look for "Private Networking" → there will be a hostname like `letta-2eot.railway.internal` or `letta-abcd.railway.internal`. **Write this down** — you'll need it for `LETTA_BASE_URL` on the api-server.

## 5. Wire the api-server (`capabilityeconomics`)

Open the `capabilityeconomics` service → **Variables** tab → add or update these:

```env
# Mem0 — internal DNS, X-API-Key (NOT Bearer)
MEM0_BASE_URL=http://mem0.railway.internal:8000
MEM0_API_KEY=<MEM0_ADMIN_API_KEY — the same value you set as ADMIN_API_KEY on Mem0>

# Letta — note the suffix from step 4 step 6, and port 8080 (Letta's internal default with this image)
LETTA_BASE_URL=http://<letta-internal-hostname>.railway.internal:8080
LETTA_API_KEY=<LETTA_SERVER_PASS — same value you set as LETTA_SERVER_PASSWORD on Letta>
```

Save → api-server auto-redeploys (~2 min).

## 6. Verify the whole pipeline

Once api-server has redeployed, hit:

```
https://<your-capabilityeconomics-domain>/api/health/services
```

Expected output:

```json
{
  "overall": "ok",
  "services": [
    { "service": "mem0",   "status": "ok",  "latencyMs": 60,  "lastError": null },
    { "service": "letta",  "status": "ok",  "latencyMs": 75,  "lastError": null },
    ...
  ]
}
```

If `mem0` is `down`, look at the `lastError` — it now carries the underlying cause (DNS, connection refused, auth, etc.) thanks to the `describeError()` helper. Common errors:

| `lastError` | Means | Fix |
|---|---|---|
| `getaddrinfo ENOTFOUND mem0.railway.internal` | DNS doesn't resolve | Mem0 service is named something other than `mem0` — rename it, or change `MEM0_BASE_URL` to match |
| `connect ECONNREFUSED ...` | DNS resolves but nothing listening | Mem0 container crashed or still starting — check Mem0's runtime logs |
| `Mem0 GET … → 401: {"detail":"Invalid or expired token."}` | Wrong auth header (Bearer used) or wrong key | Confirm `MEM0_API_KEY` on api-server == `ADMIN_API_KEY` on Mem0, and verify code uses `X-API-Key` not `Bearer` |
| `Mem0 GET … → 401: {"detail":"Authentication required."}` | Auth header missing entirely | Set `MEM0_API_KEY` on the api-server |
| `mem0 timed out after 4000ms` | Mem0 reachable but slow / hung | Probably first-boot warmup — check again in 30 s. Persistent timeout = check Mem0 runtime logs for hangs in init |

If `letta` is `ok` on health but agent runs fail with `Handle … not found, must be one of []`, set `OPENROUTER_API_KEY` on the Letta service (not on api-server).

## 7. Persistence (optional, recommended for prod)

Mem0 writes a SQLite history database at `/app/history/history.db` inside the container. By default this is **ephemeral** — every redeploy starts with empty history. The vectors themselves are safe (those live in pgvector), but the audit trail of memory ops is lost.

To persist:
1. Mem0 service → Settings → **Volumes** → **Mount Volume** → mount point: `/app/history` → name it `mem0-history` → 1 GB is plenty.
2. Next deploy will have the volume mounted; the SQLite file survives restarts.

## TL;DR cheat sheet

| Step | Service | Source | Root dir | Required env (manual) |
|---|---|---|---|---|
| 2 | `pgvector` | Railway template (`pgvector/pgvector:pg18`) | — | None — auto-wired |
| 3 | `Mem0` | this repo | `mem0` | `OPENAI_API_KEY`, `ADMIN_API_KEY`, `JWT_SECRET`, `AUTH_DISABLED=false`, `HISTORY_DB_PATH=/app/history/history.db` (+ all `POSTGRES_*`) |
| 4 | `letta` | this repo | `letta` | `LETTA_SERVER_PASSWORD`, `OPENROUTER_API_KEY` |
| 5 | `capabilityeconomics` | this repo | `/` | `MEM0_BASE_URL`, `MEM0_API_KEY`, `LETTA_BASE_URL`, `LETTA_API_KEY` |

**The four traps to avoid** (covered in detail in [`railway-setup.md`](./railway-setup.md)):

1. Don't pull `mem0/mem0-api-server` from Docker Hub — arm64-only, won't run on Railway.
2. Don't point Mem0 at `mem0ai/mem0` GitHub directly — their `server/Dockerfile` is missing `libpq5`, runtime crashes on `import psycopg`. Use this repo's `mem0/Dockerfile`.
3. Don't use `Authorization: Bearer <ADMIN_API_KEY>` against Mem0 — v2.x treats Bearer as JWT. Use `X-API-Key`.
4. Don't forget `OPENROUTER_API_KEY` on the **Letta service itself** (not on api-server) — Letta needs it to register model handles.
