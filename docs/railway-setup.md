# Railway Setup — Capability Economics

How this app is hosted on Railway, what every service does, the pitfalls we tripped over getting here, and a clean recipe for spinning up a fresh environment.

> **Scope:** production hosting on Railway. For repo layout and local dev, see `CLAUDE.md`.

---

## TL;DR

The platform runs as **six Railway services** in one project:

| Service | Image / Source | What it does |
|---|---|---|
| `capabilityeconomics` | This repo, root `Dockerfile`, branch `main` | Express api-server + serves the built SPA |
| `Postgres` | `ghcr.io/railwayapp-templates/postgres-ssl:18` | App database (capabilities, orgs, sessions, etc.) |
| `Mem0` | This repo, root `mem0/`, our `Dockerfile` | Self-hosted memory layer for the autonomous agent (libpq5 patched, clones mem0ai/mem0) |
| `pgvector` | `pgvector/pgvector:pg18` | Vector store backing Mem0's embeddings |
| `letta-2EOT` | `letta/letta:latest` Docker image | Stateful agent runtime |
| `Neo4j Graph Database (Metal-Ready)` | `neo4j:5.26-community-bullseye` | Graph store (future capability graph use) |

The api-server (`capabilityeconomics`) is the only public-internet-facing service. The others communicate over Railway's private network at `<service-name>.railway.internal`.

## Service-by-service

### 1. `capabilityeconomics` — the api-server

- **Source:** This repo, branch `main`, builder is the root `Dockerfile`
- **Public:** Yes — `capabilityeconomics-staging.up.railway.app`
- **Required env:** `DATABASE_URL`, `PORT` (Railway injects), `ADMIN_API_KEY`, plus consumer keys for any integration it should talk to (see env reference below)
- **Build:** `pnpm install --frozen-lockfile && pnpm run build:deploy` → `pnpm run start`

This service does *not* host Mem0/Letta/Postgres — it only consumes them. It also serves the built `capability-economics` SPA as static files (`artifacts/capability-economics/dist/public`).

### 2. `Postgres` — app database

- Standard Railway Postgres template (SSL-enabled image)
- `DATABASE_URL` auto-wired into `capabilityeconomics`
- Schema managed via `drizzle-kit push` (no migration files — see CLAUDE.md)

### 3. `Mem0` — the part that bit us repeatedly

- **Source: this repo, root directory `mem0`** (builder picks up `mem0/Dockerfile`)
- Our Dockerfile installs `libpq5` (which mem0ai/mem0's own `server/Dockerfile` omits), then clones mem0ai/mem0 at the pinned tag and runs its server. Without the libpq5 install, Mem0 crashes on startup with `ImportError: no pq wrapper available … libpq library not found`.
- **DO NOT** set the source to `mem0/mem0-api-server` (Docker Hub image) — that image is **arm64-only** and won't run on Railway's amd64 infra; every deploy attempt hangs in `DEPLOYING` and eventually fails.
- **DO NOT** set the source directly to `mem0ai/mem0` repo's `server` dir either — that *builds* fine on amd64 but its container crashes at runtime on the libpq miss.
- **Required env on the Mem0 service:**
  - `OPENAI_API_KEY` — used for LLM extraction and embedding generation
  - `JWT_SECRET` — anything strong (`openssl rand -base64 48`)
  - `ADMIN_API_KEY` — anything strong; api-server uses this as the bearer token
  - `AUTH_DISABLED=false` (the default) — keep auth on
  - `POSTGRES_HOST=pgvector.railway.internal` — auto-wires to the pgvector service
  - `POSTGRES_PORT=5432`
  - `POSTGRES_DB=postgres`
  - `POSTGRES_USER=postgres`
  - `POSTGRES_PASSWORD=<auto-generated by Railway when pgvector was created>`
- **Internal hostname:** `mem0.railway.internal:8000`
- **Note:** Mem0's self-hosted REST API expects `Authorization: Bearer <ADMIN_API_KEY>`, **not** `X-API-Key`. Our api-server's `services/agent/memory.ts` was wrong for a long time — fixed in commit `f019d29`.

### 4. `pgvector` — Mem0's vector backend

- Standard `pgvector/pgvector:pg18` image
- Persists Mem0's embedding vectors. Without it, Mem0 has nowhere to write memory.
- `Postgres` (the app DB) and `pgvector` (the Mem0 store) are **separate** services in this project. Don't merge them.

### 5. `letta-2EOT` — stateful agent

- Image: `letta/letta:latest` (works on amd64, unlike Mem0's Docker Hub image)
- **Required env on the Letta service:**
  - `LETTA_SERVER_PASSWORD` — anything strong; api-server sends this as `LETTA_API_KEY`
  - `OPENROUTER_API_KEY` (or other provider) — **without a provider key, Letta has zero registered model handles** and any agent run fails with `NOT_FOUND: Handle <model> not found, must be one of []`. The health probe will still report `ok` because Letta's HTTP is responding, but Letta can't actually reason.
- **Internal hostname:** `letta-2eot.railway.internal:8080` (note the `-2eot` suffix and port `8080`, not the default `8283`)

### 6. `Neo4j Graph Database (Metal-Ready)` — graph store

- Image: `neo4j:5.26-community-bullseye`
- Currently deployed but **not yet wired into the api-server** — no `NEO4J_*` env vars consumed by any code today
- Intended use: mirroring the capability graph (`capabilities`, `capability_dependencies`, `capability_metrics`) for fast graph traversals

## How `capabilityeconomics` is wired to its dependencies

In the **`capabilityeconomics` service Variables tab**, these need to be set:

```env
# App DB
DATABASE_URL=<auto-wired from Postgres service>

# Self-hosted Mem0
MEM0_BASE_URL=http://mem0.railway.internal:8000
MEM0_API_KEY=<the ADMIN_API_KEY set on the Mem0 service>

# Self-hosted Letta
LETTA_BASE_URL=http://letta-2eot.railway.internal:8080
LETTA_API_KEY=<the LETTA_SERVER_PASSWORD set on the Letta service>

# AI integrations
OPENROUTER_API_KEY=<your key>
PERPLEXITY_API_KEY=<your key>

# Auth / billing / observability
ADMIN_API_KEY=<for the api-server's own admin routes — not the same as Mem0's>
CLERK_SECRET_KEY=<your key>
STRIPE_SECRET_KEY=<your key>
STRIPE_WEBHOOK_SECRET=<your key>
```

Verify everything with:

```
GET https://capabilityeconomics-staging.up.railway.app/api/health/services
```

Each service entry returns `ok` / `degraded` / `down` / `not_configured`. The `not_configured` state means an env var is intentionally absent — that's fine, the integration just turns off. `down` / `degraded` are real problems.

## The pitfalls we hit (so you don't again)

### 1. **`mem0/mem0-api-server:latest` is arm64-only**

- The Docker Hub image at `mem0/mem0-api-server` ships only an arm64 manifest. Railway runs on amd64.
- Railway's own Mem0 template page calls this out:
  > *"The `mem0/mem0-api-server` image on Docker Hub is arm64-only, which is incompatible with Railway's amd64 infrastructure."*
- Symptom: every Mem0 deploy hangs in `DEPLOYING` for ~15+ min then dies. api-server health probe shows `fetch failed` (DNS resolves but nothing's listening).

### 2. **mem0ai/mem0's `server/Dockerfile` is missing `libpq5`**

Even after avoiding pitfall #1 by building from the mem0ai/mem0 GitHub source directly, the upstream `server/Dockerfile` pip-installs `psycopg` (the Postgres driver) but doesn't `apt install libpq5`. Result: the build succeeds, the image is `linux/amd64`, the container deploys with status `SUCCESS`… and then crashes on startup with:

```
ImportError: no pq wrapper available.
- couldn't import psycopg 'python' implementation: libpq library not found
```

Status flips `SUCCESS → CRASHED` within ~5s of boot, restart-loops, then api-server health probe shows `timed out after 4000ms` (container is alive long enough to accept a TCP handshake but dies before responding).

**The fix:** the `mem0/Dockerfile` in this repo wraps mem0ai/mem0's build with the missing `apt install libpq5` step (plus `libpq-dev` + `build-essential` for pip-install robustness), then clones the upstream repo at a pinned release tag. Always point Railway's Mem0 service at *our* `mem0/` directory, not at the mem0ai/mem0 repo directly.

### 2. **Our api-server was sending the wrong Mem0 auth header**

- Old code: `X-API-Key: <key>` (commit `bba283a` and earlier)
- Correct: `Authorization: Bearer <key>` (commit `f019d29`)
- This was silently fine when `AUTH_DISABLED=true` on Mem0, but would 401 the moment auth was turned on. We fixed it preemptively.

### 3. **"`fetch failed`" hides the real error**

- Node.js's `fetch` wraps DNS / TCP / TLS failures into a single useless top-level message `"fetch failed"` and stashes the real cause (ENOTFOUND, ECONNREFUSED, etc.) in `err.cause`.
- Our `services/health/probes.ts` was only capturing `err.message`, so we saw `fetch failed` for *every* failure mode.
- Fixed in commit `4118cc7` — `describeError()` walks the cause chain. The probe now reports `fetch failed → getaddrinfo ENOTFOUND mem0.railway.internal`, which tells you in one line whether it's DNS, connection refused, or a timeout.

### 4. **Letta health "ok" doesn't mean Letta works**

- The health probe just checks if Letta's HTTP responds. It does.
- But if `OPENROUTER_API_KEY` (or another provider key) isn't set on the Letta service, Letta has zero registered LLM handles, and every agent run fails with `NOT_FOUND: Handle openrouter/anthropic/claude-3.7-sonnet not found, must be one of []`.
- **The fix:** set `OPENROUTER_API_KEY` on the Letta service itself (not on `capabilityeconomics`).

### 5. **Letta's internal port is 8080, not the default 8283**

- The official `letta/letta:latest` image's internal port differs from the default documented elsewhere
- `LETTA_BASE_URL` on `capabilityeconomics` must be `http://letta-2eot.railway.internal:8080`

### 6. **Replit Secrets only propagate to processes started *after* the secret is set**

- If you add `RAILWAY_TOKEN` (or anything else) to Replit Secrets while a shell is running, the running shell does NOT pick it up.
- For an already-running session, the only way to authenticate the Railway CLI is `railway login --browserless` from a Replit Shell tab (a real TTY).

### 7. **Don't paste tokens in chat**

- The `!` prefix only triggers shell execution when typed *directly into the Claude Code prompt*. Pasting `! <cmd>` as a regular message leaves the token in the chat transcript.
- Use either Replit Secrets or `railway login --browserless` from a Shell tab.

### 8. **Railway dashboard source changes don't always auto-trigger a deploy**

- If you change a service's source (e.g. Docker image → GitHub repo), Railway *sometimes* auto-deploys, sometimes not.
- If after a source change `railway deployment list --service <name>` doesn't show a new deployment within ~30s, click **Deploy** manually in the dashboard.

## Spinning up a fresh environment (recipe)

Order matters — services that other services depend on must come up first.

1. **Create a new Railway project**. Call it whatever.

2. **Add Postgres** (template). Railway will auto-wire `DATABASE_URL` into the rest of the project.

3. **Add pgvector** (template — search for "pgvector" in the Railway template gallery, image `pgvector/pgvector:pg18`).

4. **Add Mem0**:
   - New Service → "Deploy from GitHub repo" → **this repo** (`smithdouglas404/Capability-Economics-Guide`)
   - Settings → Source → **Root Directory: `mem0`** (so Railway finds `mem0/Dockerfile`, which wraps mem0ai/mem0 with the missing `libpq5` install — see pitfall #2 below for why)
   - Variables → set `OPENAI_API_KEY`, `JWT_SECRET`, `ADMIN_API_KEY`, `AUTH_DISABLED=false`, plus the `POSTGRES_*` set pointing at the pgvector service
   - First deploy takes ~5 min (apt + clone + pip install)
   - **Re-numbering:** the rest of this doc references "pitfall #1", "pitfall #2" — these are the numbered pitfalls in the **Pitfalls** section above, not these steps.

5. **Add Letta**:
   - New Service → "Docker Image" → `letta/letta:latest`
   - Variables → `LETTA_SERVER_PASSWORD=<strong>`, `OPENROUTER_API_KEY=<your key>`
   - Note the internal hostname Railway assigns (Settings → Networking)

6. **Add `capabilityeconomics`** (the api-server):
   - New Service → GitHub repo → this repo, branch `main`, root `/`
   - Railway auto-uses the root `Dockerfile`
   - Variables — set everything listed in the "How `capabilityeconomics` is wired" section above
   - Generate a public domain (Settings → Networking → Generate Domain)

7. **Push schema to the new Postgres**:
   ```bash
   DATABASE_URL=<new-prod-url> cd lib/db && npx drizzle-kit push --force
   ```

8. **Verify**:
   - `GET <public-url>/api/health/services` — all services should be `ok` or `not_configured`
   - The degraded-banner on the SPA should be absent

## Useful Railway CLI commands

Assumes you've done `railway login` and `railway link` against this project.

```bash
# State
railway status                                   # current project/env/service link
railway service status --all                     # status of every service
railway variables --service <name>               # list env vars (name=value)
railway variables --service <name> --kv          # K=V format

# Logs
railway logs --service <name> --build --since 5m # build logs (non-streaming, fetch last 5 min)
railway logs --service <name> --since 5m         # deploy/runtime logs
railway logs --service <name> --http --lines 50  # HTTP access logs

# Deployments
railway deployment list --service <name>         # last N deployments
railway service status --service <name>          # current deployment + status
railway redeploy --service <name>                # re-run latest deployment
railway down --service <name>                    # remove most recent deployment

# Trigger a deploy from a specific git SHA via GraphQL (when CLI's redeploy isn't enough):
TOKEN=$(jq -r .user.accessToken ~/.railway/config.json)
curl -X POST https://backboard.railway.app/graphql/v2 \
  -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"query":"mutation { serviceInstanceDeployV2(serviceId: \"<id>\", environmentId: \"<id>\", commitSha: \"<sha>\") }"}'
```

## When something breaks

1. **First stop:** `GET /api/health/services`. The `lastError` field on each service now carries the underlying cause (DNS, connection refused, auth, etc.) thanks to commit `4118cc7`.
2. **If a service shows `down` with `getaddrinfo ENOTFOUND`** → wrong hostname or service doesn't exist. Check the internal hostname on the upstream service's Settings → Networking.
3. **If `ECONNREFUSED`** → DNS resolves but no container is listening. Check the upstream service's deployment status — it's probably crashed or stuck.
4. **If `401`/`403`** → wrong auth header or wrong key. For Mem0, the api-server now sends `Authorization: Bearer`. The key on the api-server (`MEM0_API_KEY`) must equal `ADMIN_API_KEY` on the Mem0 service.
5. **If `timed out after 4000ms`** → upstream is responding too slowly. Probably warming up; check again in 30s. If persistent, the upstream is hung in init — check its runtime logs.
